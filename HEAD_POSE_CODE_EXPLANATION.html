1. Overview

What This Module Does

This module estimates head pose (yaw, pitch, roll angles) from facial landmarks detected by MediaPipe. It determines how a person's head is oriented in 3D space based on a 2D camera image.

The module has two methods:

  1. Primary Method: Uses MediaPipe's pre-computed transformation matrix (smooth and preferred)
  2. Fallback Method: Uses OpenCV's solvePnP algorithm when the matrix isn't available

Understanding Head Pose Angles

  • Yaw: Turning head left/right (like shaking your head "no")
  • Pitch: Tilting head up/down (like nodding "yes")
  • Roll: Tilting head sideways (like touching your ear to your shoulder)

7. Camera Intrinsics Matrix

Lines 96-104: Building the Camera Matrix

// Camera intrinsics (approx) - EXACT same as Timothy's const fx = width, fy = width, cx = width / 2, cy = height / 2; const cameraMatrix = cv.matFromArray(3, 3, cv.CV_64F, [ fx, 0, cx, 0, fy, cy, 0, 0, 1 ]); const distCoeffs = cv.Mat.zeros(4, 1, cv.CV_64F);

Code Explanation

  • fx, fy: Focal length (how "zoomed in" the camera is). Using image width as an approximation.
  • cx, cy: Principal point (where the optical axis hits the image). Assumed to be image center.
  • distCoeffs: Lens distortion coefficients, set to zero (assuming no distortion).

The camera matrix has this structure:

| fx  0  cx |
| 0  fy  cy |
| 0   0   1 |

Intuition: What is the Camera Matrix?

The camera intrinsics matrix describes the internal properties of the camera — how it projects 3D points onto the 2D image sensor. Think of it as encoding:

  • Focal length (fx, fy): How much the camera "zooms". Larger values = more zoomed in.
  • Principal point (cx, cy): Where the center of the lens projects onto the image (usually the image center).

Since we don't know the actual webcam specifications, we use reasonable approximations. This works well enough because small errors in intrinsics mainly affect translation, not rotation angles.

8. Solving the PnP Problem

Lines 110-114: The Core Algorithm

// Use SOLVEPNP_ITERATIVE without extrinsic guess - exactly like Timothy const ok = cv.solvePnP( modelPoints, imagePoints, cameraMatrix, distCoeffs, rvec, tvec, false, cv.SOLVEPNP_ITERATIVE );

Code Explanation

This is the core computer vision algorithm. Parameters:

  1. modelPoints: 3D coordinates of face model (input)
  2. imagePoints: 2D pixel coordinates from image (input)
  3. cameraMatrix: Camera intrinsics (input)
  4. distCoeffs: Distortion coefficients (input)
  5. rvec: Rotation vector (output)
  6. tvec: Translation vector (output)
  7. false: Don't use previous values as initial guess
  8. SOLVEPNP_ITERATIVE: Use iterative Levenberg-Marquardt optimization

Returns true if successful, false otherwise.

Intuition: What is PnP?

PnP (Perspective-n-Point) is a classic computer vision problem:

"Given n correspondences between 3D points and their 2D projections, plus camera parameters, find the pose (rotation + translation) of the object."

The algorithm finds the rotation and translation that, when applied to the 3D model points and projected through the camera, best matches the observed 2D image points.

The SOLVEPNP_ITERATIVE method uses Levenberg-Marquardt optimization to refine the solution iteratively for higher accuracy.

Intuition: Rotation Vector vs. Rotation Matrix

A rotation vector (rvec) is a compact 3-element representation where:

  • The direction of the vector is the axis of rotation
  • The magnitude (length) is the angle of rotation in radians

A rotation matrix is a 3×3 matrix that directly encodes how coordinate axes transform. The Rodrigues formula converts between these representations.

We need the matrix form to extract human-readable Euler angles (yaw, pitch, roll).

9. Extracting Euler Angles

Lines 130-146: Z-Y-X Euler Angle Decomposition

// Z-Y-X Euler (yaw, pitch, roll) - EXACT same as Timothy's const RAD2DEG = 180 / Math.PI; const r20 = Rjs[2][0], r00 = Rjs[0][0], r10 = Rjs[1][0]; const r21 = Rjs[2][1], r22 = Rjs[2][2]; const r01 = Rjs[0][1], r11 = Rjs[1][1]; let yaw = Math.asin(Math.max(-1, Math.min(1, -r20))); const cosYaw = Math.cos(yaw); let roll, pitch; if (Math.abs(cosYaw) > 1e-6) { roll = Math.atan2(r10, r00); pitch = Math.atan2(r21, r22); } else { roll = Math.atan2(-r01, r11); pitch = 0; }

Code Explanation

  1. RAD2DEG: Conversion factor from radians to degrees
  2. Extract specific elements from the rotation matrix needed for Euler decomposition
  3. Yaw = arcsin(-R[2][0]), clamped to [-1, 1] for numerical safety
  4. Check if we're in gimbal lock (cosYaw ≈ 0):
    • Normal case: Use standard formulas for roll and pitch
    • Gimbal lock: Use alternate formulas to avoid division by zero

Intuition: Why These Formulas?

A rotation matrix R can be decomposed into three sequential rotations (Euler angles). For Z-Y-X convention (common in aerospace):

R = R_z(yaw) × R_y(pitch) × R_x(roll)

The formulas come from inverting this composition. The key relationships are:

  • yaw = arcsin(-R[2][0])
  • roll = atan2(R[1][0], R[0][0])
  • pitch = atan2(R[2][1], R[2][2])

Gimbal lock occurs when yaw = ±90°. At this point, roll and pitch axes align, and we lose one degree of freedom. The code handles this by using an alternate formula.

The Math.max(-1, Math.min(1, ...)) clamping prevents errors from numerical precision issues (asin requires input in [-1, 1]).

10. Angle Adjustments & Conventions

Lines 148-169: Convention Adjustments

// Timothy's pitch adjustment if (pitch < 0) { pitch = pitch * RAD2DEG + 180; } else { pitch = pitch * RAD2DEG - 180; } yaw *= RAD2DEG; roll *= RAD2DEG; // Timothy's roll wrap if (roll < -90 && roll >= -180) { roll = 180 + roll; } else if (roll > 90 && roll < 180) { roll = -180 + roll; } // Timothy's final output angles = { yaw: yaw, pitch: pitch * -1, roll: roll * -1 }; R.delete(); }

Code Explanation

  1. Pitch adjustment: Add/subtract 180° to flip the coordinate convention:
    • Looking up = positive pitch
    • Looking down = negative pitch
  2. Convert to degrees: Multiply yaw and roll by RAD2DEG
  3. Roll wrapping: Keep roll in a continuous range near 0° for typical head positions (avoids jumps at ±180°)
  4. Final sign flips: Multiply pitch and roll by -1 to match expected conventions
  5. Memory cleanup: Delete the rotation matrix to free WebAssembly memory

Intuition: Why All These Adjustments?

Different applications use different conventions for what "positive" and "negative" angles mean. These adjustments ensure:

  • Yaw: Turning left = negative, turning right = positive
  • Pitch: Looking up = positive, looking down = negative
  • Roll: Tilting right = positive, tilting left = negative

The roll wrapping prevents the angle from jumping discontinuously when the head tilts past certain thresholds (e.g., from +179° to -179°).

11. Focal Length Estimation Helper

Lines 203-233: estimateFocalLengthFromFaceSize()

/** * Estimate focal length from face size in the image */ function estimateFocalLengthFromFaceSize(landmarks, imageWidth) { if (!landmarks || landmarks.length < 468) { return null; } try { const leftEye = landmarks[33]; const rightEye = landmarks[263]; if (!leftEye || !rightEye) { return null; } const eyeDistancePixels = Math.abs(rightEye.x - leftEye.x) * imageWidth; const AVERAGE_IPD_MM = 63; const TYPICAL_DISTANCE_MM = 550; const estimatedFocalLength = (eyeDistancePixels * TYPICAL_DISTANCE_MM) / AVERAGE_IPD_MM; if (estimatedFocalLength < imageWidth * 0.3 || estimatedFocalLength > imageWidth * 2.0) { return null; } return estimatedFocalLength; } catch (error) { console.error('Error estimating focal length:', error); return null; } }

Code Explanation

  1. Get the left and right eye corner landmarks (indices 33 and 263)
  2. Calculate the distance between eyes in pixels
  3. Use known values:
    • AVERAGE_IPD_MM = 63: Average human inter-pupillary distance (63mm)
    • TYPICAL_DISTANCE_MM = 550: Typical sitting distance from screen (55cm)
  4. Apply the pinhole camera formula to estimate focal length
  5. Sanity check: reject values outside reasonable range (0.3× to 2× image width)

Intuition: The Pinhole Camera Model

The relationship between real-world size, image size, distance, and focal length is:

focal_length = (pixel_size × real_distance) / real_size

If we know:

  • Human eyes are ~63mm apart (real_size)
  • User is ~550mm from camera (real_distance)
  • Eyes appear X pixels apart in image (pixel_size)

We can estimate the camera's focal length. This is useful for applications needing more accurate camera parameters.
